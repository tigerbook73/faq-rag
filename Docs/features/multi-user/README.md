# Multi-user

## 恢复入口

当用户说“继续 multi-user 的开发”或 “continue multi-user” 时，从这里开始。

按顺序阅读：

1. `PROGRESS.md`
2. 旧版设计文档：`../../DESIGN-multi-user.md`
3. 旧版需求文档：`../../FEATURE-multi-user.md`

当前 feature 还没有完全迁移到拆分文档结构。在迁移提交完成前，以上旧版文档仍然是需求和设计的事实来源，本目录用于恢复进度和记录实施状态。

## 文档职责

- `PROGRESS.md`：当前实施状态、历史提交映射、已知不一致、验证状态和下一步入口。
- `../../DESIGN-multi-user.md`：技术设计、实施阶段、测试策略和发布策略。
- `../../FEATURE-multi-user.md`：产品需求、用例、业务规则和验收标准。

## 恢复规则

- 不要一开始就扫描整个代码库。
- 先阅读 `PROGRESS.md`，再检查 `git status` 和最近提交。
- 只有当进度文档、提交记录或当前 diff 不足以判断真实状态时，才检查代码。
- 如果旧版需求或设计文档在最后确认进度之后发生变化，并且变化会影响实施流程，先暂停编码，刷新 feature 文档后再继续。
- 当阶段或子阶段提交会改变实施状态、下一步、已知不一致或验证结果时，提交前更新 `PROGRESS.md`。
