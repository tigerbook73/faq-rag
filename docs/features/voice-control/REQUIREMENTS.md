# Voice Control — Requirements

## 产品目标

允许用户通过自然语言（含语音输入）控制 FAQ RAG 系统，涵盖核心问答流程以及文档管理、会话管理等系统操作，降低键盘依赖，提升交互效率。

---

## 用户角色

| 角色 | 主要场景 |
|------|---------|
| **普通用户** | 语音提问、管理自己的文档和会话 |
| **管理员** | 在普通用户场景之上，语音控制用户/文档管理 |

---

## 核心用例

### UC-1：语音提问（最高优先级）
用户按住/点击麦克风按钮，说出问题，系统识别为文字后发送给 RAG 检索引擎，返回带引用的流式回答。
- 与现有文字问答体验一致，只是输入方式不同
- 需要支持中英文混合输入

### UC-2：自然语言管理会话
- "新建一个会话" → `POST /api/sessions`
- "删除这个会话" / "清空对话" → `DELETE /api/sessions/:id`
- "切换到上一个会话" → 前端路由

### UC-3：自然语言管理文档
- "列出我的文档" → `GET /api/documents`
- "把《XX》设为公开" → `PATCH /api/documents/:id`（visibility）
- "删除《XX》" → `DELETE /api/documents/:id`（需确认）
- "重建《XX》的索引" → `POST /api/documents/:id/reindex`

### UC-4：切换 LLM 提供商
- "切换到 Claude" / "用 DeepSeek 回答" → 前端 provider 状态变更

### UC-5：管理公共文档（管理员）
- "添加《XX》到我的知识库" → `POST /api/public-documents/:id/selection`
- "查看可用的公共文档" → `GET /api/public-documents`

## 业务规则

1. **单一输入入口**：不区分"问答模式"和"命令模式"，由系统自动判断用户意图
2. **结果连续呈现**：命令执行结果以对话气泡形式插入聊天流，不弹窗打断体验
3. **危险操作需确认**：删除类操作须二次确认（语音或文字均可回应"是" / "确认"）
4. **权限继承**：语音命令受与文字操作相同的权限控制（普通用户不能执行管理员命令）
5. **优雅降级**：STT 识别失败时，回退到文字输入，不影响正常使用
6. **隐私**：语音数据不持久化存储，仅用于实时识别

---

## 可控范围边界

### 在范围内
- 对话问答（UC-1）
- 会话 CRUD（UC-2）
- 文档查看 / 删除 / 可见性 / 重建索引（UC-3）
- 提供商切换（UC-4）
- 公共文档选择（UC-5）

### 超出范围（本功能不涵盖）
- 文件上传（需要文件选择器，语音仅能触发打开，无法完成上传）
- 创建新用户账号（多步骤参数，复杂度高）
- 重置密码（安全敏感，不适合语音）
- 唤醒词检测（"Hey FAQ"）
- 离线语音识别
- TTS（文字转语音反馈）— 初期不包含，后续可按需扩展

## 验收标准

| 编号 | 验收条件 |
|------|---------|
| AC-1 | 用户点击麦克风按钮后可录音，识别结果自动填入输入框并发送 |
| AC-2 | 中文、英文、中英文混合输入均可正确识别和处理 |
| AC-3 | 说出系统命令（如"删除文档"）可正确触发对应 API 调用 |
| AC-4 | 删除等危险操作在执行前出现确认对话 |
| AC-5 | STT 识别失败时给出友好提示，不影响其他功能 |
| AC-6 | 命令执行结果以对话消息形式呈现，无跳出感 |
| AC-7 | 未登录用户无法使用语音功能，权限错误有提示 |
